敏捷开发助力平安保险快速推出创新业务模式
------平安集团P2P项目
概 述
      2011年，中国平安保险(集团)股份有限公司计划推出P2P网络借贷平台，作为一种新的业务模式，该平台是为个人投资者和贷款者提供的网上投融资平台，为双方提供信息查询、验证、传递服务，以及评估交易的风险状况和利率水平，帮助实现交易并减少风险，是平安集团创新金融业务试点。
      该模式下，通过P2P网络借贷平台，借款人可以向担保公司提出贷款申请，投资人可以查看并选择合适的贷款请求进行投资。投资双方在平台上达成一致之后通过平台来按期还款／收款，平台提供利息计算，分期还款等功能。
      由于该平台涉及在线业务、金融分析、利息计算等业务，对系统稳定性及质量提出了很高的要求。同时，作为平安的创新金融业务，对业务流程的梳理及快速的在系统中进行实现，快速推出面向市场都有极高的要求。

面临的问题与挑战
•	创新的业务模式，业务人员无法清楚描述具体的业务流程和要求，如何准确的开发出客户心中所希望的系统？
•	相关人员职责不清晰，无法有效配合项目开展，参与度很低，如何快速让业务人员理解当前项目情况及提出合理的意见和建议？
•	作为保险行业的应用系统，相关算法和规则非常复杂，如何快速并有效的验证系统的计算结果？
•	业务部门和IT部门隶属于不同的子公司，存在一定的交流协作困难，如何提升沟通效率？
•	该项目需要融入平安整体IT环境中，如何与其他系统进行整合、联调？
•	项目周期只有三个月，就要内部上线，如何保证按时交付？




敏捷致胜
      根据P2P项目实际情况，ThoughtWorks组建了一个8个人的敏捷软件交付团队，在客户现场进行系统开发与实施，通过采用敏捷软件工程的相关实践，成功的交付了高质量的P2P网络借贷平台。
 
•	敏捷需求开发与管理
      通过项目初期的Inception workshop, 快速的将存在于业务人员脑中的流程与要求展现出来，进行深入讨论分析，获取尽可能真实有用的需求，最终在业务人员、IT部门管理人员和项目实施团队达成一致。
      Inception是一种敏捷需求开发方法，通过与业务人员在一段时间内集中讨论，并将想法具体呈现出来，用以快速的获得反馈，辅以“角色模拟”、“头脑风暴”等其他实践，能够有效的将无法说清楚的需求条理化、逻辑化呈现出来。
      在平安为期2周的Inception Workshop中，我们通过各种沟通实践，快速统一团队中各个成员对P2P产品概念，描绘到了白板上，并转化成交付计划，定下了P2P业务的方向，精确的确定了最简单系统的交付里程碑。
 
      与传统需求管理的比较：
角度	平安传统需求管理	敏捷需求管理
需求分析时机	更多集中在项目早期	除了开始的两周Quick Start外，近乎均匀地贯穿与项目的整个生命周期
需求划分单位	基于功能分解，划分模块或子系统，一个模块或子系统的颗粒度通常较大	基于是否有独立业务价值，切割成一个个用户故事，每个故事都是端到端，往往跨越模块或子系统的边界。
需求文档要求	正式文档，往往有明确格式要求。难维护，难跟踪，难阅读	非正式文档，仅用于开发团队和客户沟通。更多通过自动化功能测试用例来跟踪系统需求。
需求传递过程	单向的陈述与记录，文档传导（线性传递，误导放大，缓慢）	聚议，开发和客户共同参与，及时的非正式沟通
应对需求变更	有严格的控制流程，视变更为风险	视变更为必然或意料之中


•	快速的短周期迭代
      在项目实施期间，由于P2P项目周期时间较短，项目团队以1周为周期进行迭代，在每个迭代的最后，每个迭代的交付物都是端到端的，可测试，可验收的工作软件，便于客户能够尽早的使用已经完成的功能并提出建议，以此保证所开发的功能符合客户预期，若有变更，亦不会有较大风险。，每个迭代都由Story组成，可以根据业务需求变化做调整，但总体工作量需保持不变。
	在一个迭代内，通过固定时间、固定工作内容、制定迭代计划、每日站立会议、迭代评审会议、迭代回顾会议等实践，能够保证项目团队按时高质量的交付承诺完成的功能，客户有合理的预期，为项目平稳开展提供保证。
•	完整可用的持续集成环境
      项目实施团队在项目初期就构建了完整的持续集成环境，通过持续集成环境，能够保证每次提交至代码库的代码都可用正确编译，并通过自动化测试，结合代码质量检测机制，有效的保证项目代码质量和开发质量，为项目的高质量交付提供了基础。持续集成机制也是项目在5个月内3次发布的重要保证。
      持续集成是敏捷软件工程的重要实践，由配置管理、构建过程、部署过程、静态代码检测、自动化测试、质量报表等部分组成。通过持续集成，项目实施团队能够快速得到质量反馈，在第一时间发现问题，解决问题。
      在配置管理中，我们使用Git作为配置管理工具，主要考虑到Git具有以下优势：
o	版本库本地化，支持离线提交，相对独立不影响协同开发。每个开发者都拥有自己的版本控制库，在自己的版本库上可以任意的执行提交代码、创建分支等行为。例如，开发者认为自己提交的代码有问题？没关系，因为版本库是自己的，回滚历史、反复提交、归并分支并不会影响到其他开发者。
o	更少的“仓库污染”。Git对于每个工程只会产生一个.git目录，这个工程所有的版本控制信息都在这个目录中，不会像SVN那样在每个目录下都产生.svn目录。
o	把内容按元数据方式存储，完整克隆版本库。所有版本信息位于.git目录中，它是处于你的机器上的一个克隆版的版本库，它拥有中心版本库上所有的东西，例如标签、分支、版本记录等。
o	支持快速切换分支方便合并，比较合并性能好。在同一目录下即可切换不同的分支，方便合并，且合并文件速度比SVN快。
o	分布式版本库，无单点故障，内容完整性好。内容存储使用的是SHA-1哈希算法。确保代码内容的完整性，确保在遇到磁盘故障和网络问题时降低对版本库的破坏。
      通过持续不断的集成和发布，提升项目团队对系统的信心，基本可以做到随时提供一个可用的版本。通过随时发布，业务人员能够随时看到项目最新的进展，提出合理的调整意见，为项目向正确的方向前进保驾护航。
      通过自动化测试，保证每一个功能的开发都得到预先验证，减少后续的人工测试工作量，对重复性的测试工作由自动化测试脚本完成，提高软件质量。

Martin Fowler
我们的首席科学家Martin Fowler先生是当今世界软件开发领域最具影响力的五位大师之一。作为敏捷软件开发方法的早期开拓者和敏捷宣言的创始人之一，Fowler先生对IT 业的影响是不可估量的。
在Fowler先生的职业生涯中，他大力倡导业内最先进的软件开发技术，如统一建模语言UML、极限编程XP、重构与分析模式等。在为倡导改革的大型公司们提供解决方案的同时，他进一步完善了这些技术。这些大公司包括花旗银行、埃森哲咨询公司、Sterling Software及戴姆勒－克莱斯勒汽车公司等。
Fowler先生拥有丰富的基于对象的和基于组件的先进开发技术经验，他使用先进的技术为企业客户解决了许多实际的商业困难。作为首席科学家，Fowler先生曾经并将继续努力帮助客户开发和实施依据自己专长所开拓的最佳实践方法、领导ThoughtWorks利用最先进的技术在软业开发领域不断创新，并坚持不懈地在软业界的各种会议上推广先进的理论和技术。
Fowler先生的著作精品包括：
•	《重构－改善既有代码的设计》(Refactoring: Improving the Design of Existing Code) 。Fowler先生在本书中总结了对代码进行重构的理论与实践，指导着无数软件团队与开发者的日常工作。
•	《UML精粹：标准对象建模语言简明指南》(UML Distilled：A Brief Guide to the Standard Object Modeling) 。本书内容以UML 2.0为基础，引进交互概观图、定时图、复合结构图等新图型，深入浅出地介绍了UML的使用方法。
•	《分析模式：可重用的对象模型》(Analysis Patterns：Reusable Object Models)。 本书讲述各种分析模式（即来自概念性业务模型的模式）和支持模式（即讲述如何使用分析模式的辅助性模式），把论述重点放在介绍面向对象分析和设计的最终结果—即模型本身。作者透过平实朴素的语言，将自己丰富的对象建模经验与读者分享，使读者可以马上采纳这些经验性模式。
•	《企业应用架构模式》(Patterns of Enterprise Application Architecture)。尽管技术本身存在变化——从Smalltalk 到CORBA，再到Java和.NET，但基本的设计思想并没有太多变化，可以加以适当调整，用来解决那些共同的问题。在一组专家级合作者的帮助下，作者 将40多种经常出现的解决方案转化成模式，最终写成这本能够应用于任何一种企业应用平台的、关于解决方案的、不可或缺的手册。《企业应用架构模式》曾于 2002年荣获美国软件开发杂志图书类的生产效率奖和读者选择奖。
